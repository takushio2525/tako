# concept.md — tako のコンセプトと競合分析

> tako が「何であり、何でないか」を定義する最上位ドキュメント。
> 機能の詳細は `requirements.md`、実現手段は `architecture.md` を参照。

## 一行定義

**tako は AI 駆動・エージェント集約監視に特化した OSS の GUI ターミナルである。**
iTerm2 の「ターミナルとしての完成度」と Zed の「ネイティブ高速・モダン UI」を
足した思想で、Zed 級に高速・軽量であることを譲らない。

- 名前: tako（蛸）。多数の腕（ペイン・エージェント）を 1 つの頭（タブ）で操るイメージ
- ライセンス: Apache-2.0
- プラットフォーム: macOS 先行、**Windows 対応は必須要件**（Linux は将来検討）

## 解決する課題

AI エージェント（Claude Code 等）を使う開発では、1 つの作業が
「エージェント本体 + 子エージェント + dev サーバー + ログ監視」のように
複数プロセスへ自然に分裂する。既存ターミナルではこれらが
別タブ・別ウィンドウに散らばり、**「いまどのエージェントが何をしているか」の
全体把握コストが作業の主要なボトルネック**になっている。

tako はこれを「**1 グループ = 1 タブ**」の原則で解決する。

## コンセプト①: エージェント集約監視（コア価値）

- 通常のターミナルとして分割（split）とタブを持つ
- タブ内で動く AI エージェントが子エージェントや dev サーバーを起動すると、
  **同じタブ内に自動でペインが生える**
- エージェントはペインを生やすだけでなく、**画面構成そのものを自由に操作できる**:
  作業が終わった分割ターミナルを自分で片付ける、成果物を見せるときにプレビューペインを
  大きく開いてレイアウトを整える、など（方針:《AI での開発がやりやすく、かつ AI が
  開発をアシストできる》。操作カタログは `requirements.md` の FR-2.5）
- 結果、1 つの作業グループ（エージェント一味）を 1 タブで集約監視できる

これを実現する検知・制御は 3 層構造（詳細は `architecture.md`）:

| 層 | 仕組み | 性格 |
|---|---|---|
| Layer 1 | 汎用 CLI（`tako split` / `tako send` / `tako focus` 等）。アプリ内ターミナルに `TAKO_PANE_ID` 等の環境変数を注入し、呼び出し元ペインを自動特定 | 明示的・スクリプタブル。tmux 系オーケストレーター（spawn-worker.sh 等）との連携口 |
| Layer 2 | **内蔵 MCP サーバー**。アプリ内で起動したエージェントが環境変数経由で自動発見し、設定ゼロでペイン操作できる | **最大の差別化点**。ユーザーは何も設定しない |
| Layer 3 | パッシブ検知。OSC 7/133 シェル統合 + 子プロセスの listen ポート検知で「プレビューを開く？」の提案チップを出す | 暗黙的・opt-in。**強制分割はしない** |

### 設計原則（最優先）

**一般ユーザーが何の設定もなく使えること。**
tmux ベースのオーケストレーター構成との連携は Layer 1 CLI 経由で実現する想定だが、
それはパワーユーザー向けの拡張口であって前提ではない。
「インストールして開いたら、Claude Code を起動するだけで集約監視が動く」が基準線。

## コンセプト②: 軽量 IDE 的ワークスペース

- **左サイドバー**: cwd 連動のファイルツリー（VSCode 風）。
  ファイルクリックでペイン分割し、コード / Markdown / PDF のプレビューと軽い編集
- **右サイドバー**: git graph（コミットグラフ・ブランチ可視化）

エディタの置き換えは目指さない。「エージェントの成果物をその場で確認し、
軽く直す」までが守備範囲で、本格的な編集は外部エディタに委ねる。

## 競合分析

| 製品 | スタック | OS | ライセンス | 強み | tako から見た弱み |
|---|---|---|---|---|---|
| **cmux**（21.7k ★） | libghostty + Swift | macOS 専用 | **GPL-3.0** | コンセプト①をほぼ実装済み | macOS 専用、コンセプト②なし |
| **Wave Terminal** | Electron | クロス | Apache-2.0 | ブロック UI・プレビュー機能 | Electron で重い |
| **Warp** | Rust | クロス | OSS 化済み | 完成度・知名度 | AI がクラウド縛り（ローカルエージェント集約ではない） |
| iTerm2 / Ghostty 等 | 各種 | — | — | ターミナルとしての完成度 | エージェント集約の概念がない |

### ⚠️ cmux に関する絶対ルール

cmux は GPL-3.0 であり、tako は Apache-2.0 を採る。
**cmux のソースコードは絶対に読まない・参照しない・移植しない。**
公開されている設計思想・UX（スクリーンショット、README、ブログ等）のみを参考にする。
これはライセンス汚染を防ぐための法的要件であり、例外はない。

## 差別化軸

**「① + ② の同時実現 × ネイティブ高速 × クロスプラットフォーム × MCP 自動統合」**

- cmux に対して: クロスプラットフォーム + コンセプト② + Apache-2.0
- Wave に対して: ネイティブ（Rust + GPUI）による速度・軽さ
- Warp に対して: ローカルエージェント（Claude Code 等）との設定ゼロ統合（Layer 2 MCP）

## やらないこと（Non-goals）

- フル IDE 化（LSP 統合・デバッガ・リファクタリング支援はやらない）
- 独自 AI 機能の内蔵（AI はユーザーが選んだエージェント CLI に委ねる。tako は「器」）
- クラウド同期・アカウント機能
- tmux の代替（サーバー上の永続セッションは守備範囲外。連携はする）
