# tako remote セキュリティ・法的リスク再監査（2026-07-07）

> 対象: `tako remote` CLI / `tako_remote_*` MCP / 実装一式（`crates/tako-control/src/remote.rs`,
> `agents.rs`, `transcript.rs`, `web/tako-remote/`, `web/tako-remote-worker/`）。
> 現時点コード（main = `a762a71`）で判断。コードは変更していない（Read/Grep のみ）。

## 深刻度順の指摘一覧

### [中] LAN 直モードでは Bearer トークンが平文 HTTP でネットワークを流れる
- 場所: `remote.rs:108`（`0.0.0.0` に平文 HTTP で bind）/ `remote.rs:1112-1114`（Bearer 認証）/
  `web/tako-remote/src/api.js:9`（`Authorization: Bearer` をヘッダで送信）
- 根拠: サーバーは常に `http://0.0.0.0:7749` の平文で待受。トンネル未使用（`--no-tunnel` や
  cloudflared 不在で LAN 落ち）時、PWA は `http://<LAN-IP>:7749` に接続し、**Bearer トークンが
  リクエストヘッダに平文で載る**。同一 Wi-Fi / LAN 上の第三者はパケットキャプチャでトークンを
  取得でき、取得後は input API で任意コマンドを実行できる（= 端末奪取）。接続 URL の *fragment*
  はサーバーに送らない工夫（`remote.rs:670-718`）があるが、これは URL の話であり、確立後の
  API 呼び出しヘッダは別問題。トンネル経由なら trycloudflare が TLS 終端するため LAN 区間より安全。
- 補足: `tunnel → 127.0.0.1:port` 区間も平文だが同一ホスト内なので実害なし。

### [中] state ファイル（token / port / QR PNG）が /tmp に緩いパーミッションで置かれる
- 場所: `remote.rs:72-76`（既定 `state_dir()` = `/tmp`）/ `remote.rs:133-136`（token/port を
  パーミッション指定なしで `std::fs::write`）/ `remote.rs:1802`（QR PNG を `temp_dir` に生成）
- 根拠: `relay_secret` はわざわざ `0o600` を設定している（`remote.rs:618-622`）のに対し、
  `tako-remote.token`・`tako-remote.port`・`tako-remote-qr.png` はモード未設定 = umask 依存
  （通常 0644、他ユーザー読取可）。マルチユーザー Mac / 共有 CI では別ユーザーが `/tmp` から
  トークンや QR（トークン入り URL）を読める。QR PNG はトークンを含む接続 URL をエンコードしている。
- 対応案: token / QR も `0o600` を設定、または `state_dir()` を `data_dir()`（ユーザー専用）へ寄せる。

### [低] トークンが標準出力・IPC・UI プレビューに露出する
- 場所: `remote.rs:232-243`（起動情報 JSON に `token` を含め `println!`）/
  `main.rs:1374 remote_status`（`daemon_status` の token を `pretty_json` で標準出力）/
  CLI `tako remote start` の QR を tako-app のプレビューペインに開く（`main.rs:1330`）
- 根拠: トークンは端末画面・スクリーンショット・ターミナルのスクロールバックに残る。前回監査の
  「QR スクショにトークン写り込み」と同種の運用リスク。ログファイルには出していない（設計どおり）が、
  stdout 露出は残る。深刻度は低（ローカル画面前提）だが、画面共有・スクショ共有時に漏れる。

### [低] 認証比較が定数時間でない（タイミング攻撃の理論的余地）
- 場所: `remote.rs:1113`（`v == format!("Bearer {token}")`）/ `remote.rs:1178`（WS も `!=` 比較）
- 根拠: 文字列 `==` は不一致で早期 return しうる。ただしトークンは 256bit CSPRNG（`lib.rs:39-44`,
  `getrandom`）で網羅探索不能、かつネットワーク越しのタイミング差測定は現実的でない。実害はほぼ
  無いが、`subtle` 等の定数時間比較にすれば筋が通る。**トークン強度・生成方法自体は問題なし**。

### [低] リレー worker にレートリミットが無い
- 場所: `web/tako-remote-worker/src/index.js`（register / resolve に rate limit 記述なし）
- 根拠: 公共リレー URL は公開後に既知。UUID 形式・trycloudflare 形式・secret hex 形式の検証は
  あり、first-write-wins 保護（#78）で登録上書きは防げるが、無差別 register/resolve による
  KV 書き込み・読み取りスパム（Cloudflare 課金・KV クォータ消費）は防げない。machineId が未知なら
  resolve しても 404 のため情報漏えいはなし。作者のアカウント資源への DoS が主リスク。

## 問題なしと確認できた項目

- **トークン強度・生成**: 32 バイト CSPRNG（`getrandom::fill`）を hex 化 = 256bit。`lib.rs:39-44` で確認。
- **デーモンのトークンは起動ごとに新規生成**し停止時に破棄（`remote.rs:128`, `cleanup_state_files`）。永続トークンの垂れ流しなし。
- **/api/health 以外は全 API で Bearer 認証必須**。未認証は 401。`remote.rs:1519-1539` で確認。
- **WS も認証必須**（Sec-WebSocket-Protocol の `token.<T>`、不一致で 101 を返さない）。`remote.rs:1169-1184` で確認。
- **接続 URL のトークンは fragment に載せ**、サーバー・cloudflared ログ・Referer に平文が残らない設計。`remote.rs:670-718` + テスト `connect_urlのトークンはfragmentに載る` で確認。
- **リレーには画面内容・トークンが通らない**: KV に保存されるのは machineId（UUID）と tunnelURL のみ。`worker/src/index.js:91-100`、画面データはトンネル経由で tako 本体と直接授受。
- **リレー登録の secret は CSPRNG 生成 + 0o600 + SHA-256 ハッシュのみ worker 保存**。平文 secret は worker に渡るが KV には hash だけ。`remote.rs:598-624`, `worker/src/index.js:76-88` で確認。
- **入力バリデーション**: worker は machineId/tunnelURL/secret を正規表現で厳格検証。transcript は session_id を英数ハイフンのみに制限しパストラバーサル防止（`transcript.rs:25-33`）。pane target も抽出時に検証。
- **PWA の XSS 経路なし**: `innerHTML` / `dangerouslySetInnerHTML` の使用ゼロ（grep 一致なし）。ANSI は自前パーサ（`ansi.js`）で描画。
- **既定 off**: remote は `tako remote start` / MCP を明示実行しないと起動しない。常時待受でない。
- **tako 外での MCP ブリッジは 0 ツール**（docs 記載どおり）で、勝手なリモート公開はしない。
- **デフォルトは tunnel あり（TLS）**: cloudflared があれば LAN 平文を避けられる。

## 法的観点の整理（日本法。断定せず「リスク整理」）

> 前提: (A) 開発者 = OSS を GitHub public で配布する側、(B) 利用者 = 自分の Mac で `tako remote` を動かす側、
> (C) 作者が運用する公共リレー worker、の 3 者で扱いが変わる。

### ① 電気通信事業法
- **開発者 (A)**: ソフトウェアを配布するだけで「電気通信役務の提供」をしていない → **非該当**（事業を営んでいない）。
- **利用者 (B)**: 自分の端末を自分（または自分の管理下の端末）からリモート操作するのは「**自己の需要**に
  応ずるための通信」であり、他人の通信を媒介していない → 電気通信事業に **非該当**の可能性が高い。
  他人にアカウントを提供して媒介させる形にすると評価が変わりうる。
- **作者の公共リレー (C)**: machineId → tunnelURL の**名前解決のみ**を不特定利用者に提供する。通信内容
  （画面・入力）はリレーを通らずトンネル直通のため「他人の通信の媒介」性は弱い。ただし不特定多数へ
  継続的に役務提供する形なので、**電気通信事業（届出/登録）該当性は条件付きでグレー**。無償・付随的・
  内容非媒介という事情は非該当方向に働くが、**要専門家確認**。回避したいなら「自前リレー推奨・公共リレーは
  ベストエフォート」を明示（既に docs 記載あり）し、リレーを配布物の必須経路にしない設計を維持する。

### ② 不正アクセス禁止法（不正アクセス行為の禁止等に関する法律）
- **設計自体**: 正規トークンを持つ本人がアクセスする限り「アクセス制御機能を破る」行為ではなく **非該当**。
  トークン認証 = アクセス制御機能として機能しており、むしろ法の想定する保護側。
- **第三者がトークンを窃取して侵入した場合**: 侵入した第三者が違反主体。tako 提供者が幇助に問われる筋は
  薄い（正当な目的・正当な機能）。ただし **[中] の LAN 平文トークン漏えいを放置すると窃取を容易にする**
  ため、暗号化・パーミッション強化は「防御水準」の観点で推奨。**開発者側の違反リスクは低い**。

### ③ 個人情報保護法
- リレーが扱うのは machineId（ランダム UUID）と tunnelURL のみ。**特定個人の識別性は低く「個人情報」該当性は
  弱い**。画面内容（実名・メール等を含みうる）はリレーを通らずトンネル直通で、tako 本体のトークン認証下。
- **作者が公共リレーを運用する場合**、tunnelURL 等のログ保存を行うなら「個人情報取扱事業者」としての整理が
  必要になりうるが、現状 KV は TTL 付き machineId→URL のみで内容ログなし → **リスク低**。
  利用者 (B) が自分の画面内容を自分で扱う分には第三者提供が発生しない。**条件付き（作者のログ運用次第）**。

### ④ 外為法・輸出規制（暗号を含む OSS の public 公開）
- tako 本体は**独自の暗号アルゴリズムを実装していない**: 乱数は OS CSPRNG（`getrandom`）、TLS は
  cloudflared / OS 依存、WS マスキングは tungstenite、SHA-256 はリレー worker（Cloudflare ランタイム）。
- 公開 OSS の GitHub 配布は「**公知**の技術」「一般に入手可能なソフトウェア」に当たり、外為法の
  役務取引・輸出規制の適用除外に該当しやすい（暗号を主機能とする専用品ではない）。**リスクは低い**が、
  暗号関連 OSS 一般の論点として **専門家確認の余地**は残す（該当/非該当の断定はしない）。

### ⑤ プロバイダ責任制限法（現・情報流通プラットフォーム対処法）
- tako / リレーは「不特定の者に受信される情報の流通」（掲示板・SNS 的な公開流通）を媒介しない。リモートは
  1:1 の自己端末操作、リレーは名前解決のみ。**発信者情報開示・削除義務の対象となる典型的な場面に該当しない**
  → **非該当の可能性が高い**。

### その他の関連法（追加）
- **刑法 168 条の 2（不正指令電磁的記録＝ウイルス罪）**: 「リモートから任意コマンド実行」機能は正当な
  目的・利用者の意思に基づく操作であり、人の意思に反する不正な指令ではない → **非該当**。ただし配布時に
  「これは正規の遠隔操作ツールであり、他人の端末に無断導入する用途ではない」旨を明示すると誤解・悪用の
  抑止になる。
- **著作権法（GPL 遵守）**: 前回監査で GPL-3.0-or-later 統一済み。cloudflared / 各依存のライセンス整合は
  別途要確認だが本監査の対象外。
- **電波法・電気通信事業法の技適等**: 通信は既存 IP 網利用のため無関係。

## 推奨対応（優先順）
1. **[中]** LAN 直モードで平文トークンが流れる旨を、CLI 起動時と docs で**明示警告**する（トンネル利用を
   強く推奨、`--no-tunnel` は信頼できる LAN 限定と注記）。将来的には LAN でも自己署名 TLS / ローカル
   証明書を検討。
2. **[中]** `token_path` / QR PNG に `0o600` を設定する（`relay_secret` と同様）。可能なら state を
   `/tmp` から `data_dir()`（ユーザー専用ディレクトリ）へ移す。
3. **[低]** `remote status` のトークン出力を既定でマスクし、`--show-token` 等の明示フラグ時のみ表示。
4. **[低]** 認証比較を定数時間比較に置換（`subtle` クレート）。
5. **[低]** リレー worker に簡易レートリミット（Cloudflare の rate limiting / KV カウンタ）を追加。
6. **README にリモート機能のセキュリティ注意（任意コマンド実行・トークン管理・接続 URL の共有厳禁）を追記**
   （現状 README にリモートの記述自体が乏しい。docs には一部あり）。

  する事実は確認済み。
- リレー worker の本番デプロイ済みインスタンスの実挙動（レートリミットが Cloudflare 側でかかっているか）は
  コードのみで判断。ダッシュボード設定は未確認。

  ④ 暗号 OSS の輸出規制該当性は断定を避け、専門家判断に委ねるべき論点として明示する。
