# オーケストレーション品質設計（Issue #100）

2026-07-07。master 用デフォルトシステムプロンプトと tako setup 配布物に
「品質パイプライン」を組み込んだ設計意図の記録。対象 PR は #100 を Closes する。

## 背景: 潰したい失敗パターン

実運用（複数プロジェクト・複数モデルでのドッグフーディング）で、モデルを問わず
繰り返し観測された失敗は次の 2 つに集約される。

1. **仕事の振り分け不全** — ユーザーが 1 メッセージに複数の独立した依頼を入れると、
   master が 1 worker に全部まとめて投げる。worker の注意が分散し、各依頼の
   作り込みが薄くなり、1 件の失敗が全体の報告を巻き込む。
2. **品質の作り込み不足** — worker が「ビルドが通った」を「動く」と混同したまま
   完了報告し、master も鵜呑みでユーザーへ転送する。受け取り後にバグが発覚して
   修正のやり直しが繰り返され、ユーザーが事実上のテスターになる。

どちらも「能力」ではなく「手順の欠落」の問題。従来のプロンプトは
What（何をすべきか）は書いていたが、How（どの順に何を出力し、何を確認したら
次へ進めるか）が書かれておらず、行間を読まないモデル（Opus / Sonnet 系）では
発火しないことがあった。

## 設計原則: モデル非依存の書き方

Fable 向けの「含み」を排し、リテラル解釈でも正しく動く形に統一した。

- **番号付き手順**: 「適宜」「必要に応じて」を使わず、Step 1→2→3→4 の順序と
  「スキップ禁止」を明記する。
- **閉じた例外リスト**: 「原則 N worker に分ける。例外はこの 3 つだけ」の形。
  開いた例外（"when appropriate"）はリテラル解釈で例外が膨張する。
- **出力義務**: 「考慮せよ」ではなく「番号付きリストを書き出せ」「分担計画を
  提示せよ」「報告はこの 4 セクション」。出力物を義務にすると手順の実行自体が
  検証可能になる。
- **穴埋め式テンプレート**: worker プロンプトは自由作文ではなく型を埋める。
  空欄は削除せず `none` と書かせ、省略を可視化する。
- **禁止には理由と代替を併記**: 「まとめ投げ禁止 — 注意が分散するため。
  代わりに 1 worker = 1 成果物」。

## master プロンプトの構造（新ブロックの意図）

### `task-intake` — 失敗 1 の対策

「依頼の列挙（番号付きリスト）→ 1 worker = 1 成果物の割り当て → 並列/直列判定 →
分担計画の提示と同ターン spawn」の 4 ステップを毎メッセージ必須にした。

- 列挙を**最初の出力義務**にしたのが要点。まとめ投げは「列挙しなかった」ときに
  起きるため、列挙さえ強制すれば割り当てステップで自然に分かれる。
- 統合を許す例外は 3 つに閉じた: 同一ファイル重複（並列編集は禁止）/
  パイプライン依存（B が A の出力を要する）/ リポ変更なし（master が直接回答）。
- 逆方向の失敗（1 つの成果物をステップ単位に刻んで統合バグを作る）も明記した。
  分割の単位は「成果物」であって「実装ステップ」ではない。
- 計画提示後に承認待ちで止まるのは「途中停止」という失敗だと明記
  （破壊的操作と本当に曖昧な分割のみ確認する）。

### `worker-prompt-template` — 失敗 2 の対策（入口）

worker の品質は worker プロンプトで決まる。tako の worker は素の claude として
起動する（専用 system prompt を持たない）ため、**worker に効かせたい規律は
すべてプロンプト本文で運ぶ**必要がある。そこで master に自由作文をさせず、
8 セクションの型（Task / Background / Scope / Constraints / 受け入れ条件 /
検証手順 / Git / 報告様式）を埋めさせる。

- **受け入れ条件**は「コマンドか観察で検証可能な文」に限定。書けないなら
  タスクが未定義（ユーザーに確認するか偵察 worker を先に出す）。
- **検証手順**に「ビルドが通った ≠ 動く」「変更を実際に動かして観察」
  「diff の読み直し」を焼き込み、worker の自己検証を義務化。
- **報告様式**は 4 セクション固定で、受け入れ条件ごとの証拠（実行コマンドと
  実出力）を要求。「証拠のない Done は差し戻される」と worker 側に予告する。
- 根因先行（バグは再現・根因を Background に書いてから委任。無ければ偵察を先に）、
  要件密着タスク（課題・仕様書もの）は master が要件を転記し
  「書いてないことはやらない」を明示、の 2 ルールを型の記入規則として付けた。
- サブエージェント・バックグラウンドオーケストレーションの禁止を Constraints の
  固定文に入れた（進捗がペインに見えることが tako の設計前提のため）。

### `acceptance` — 失敗 2 の対策（出口）

worker の完了報告をユーザーへ中継する前の検収手順。

- 受け入れ条件×証拠の突き合わせ → 足りなければ「どの証拠が無いか」を 1 通で
  差し戻す（曖昧な聞き直し禁止）。
- master 自身の**的を絞った** diff スポットチェックを義務化。「A を B に」系は
  実コードで A が B になったかを確認する。従来の「master は調査しない」原則とは
  矛盾しないよう、no-investigate 側に「検収時の的を絞った検証 Read は職務」と
  例外を明記した（トークン規律は保ちつつ検収は成立させる）。
- 機械検証できない領域（UI の見た目・実機・IME・描画）は操作ログ/スクショが
  無ければ「実装済み・未検証」としてしか報告できない。
- 差し戻しは「期待 vs 実際」の欠陥リストで行い、**2 回失敗したら 3 度目の
  盲目リトライをせず**、根因かタスク分割を疑ってユーザーに現状を報告する。
- Issue の Closes 判断は master が持つ（報告環境での解消確認なしに閉じない）。

### 既存ブロックへの運用知見の反映

- `monitoring`: WORKER_IDLE 通知は空振りしうる → read_pane で確認してから行動。
  高 effort の長考は正常（目安 10 分は待つ）。close → respawn してよい条件を
  閉じたリストにした（明示的エラー / 無反応 10 分超 / worker 自身の断念宣言）。
  thinking 中の respawn はコンテキスト全損 = トークン二重消費と理由を併記。
- `lifecycle`: 高 ctx 時の解放手順に「コミット確定の確認」を追加、長尺タスクは
  マイルストーンごとのコミットをプロンプトで義務化（中断しても成果が残る）。
- `spawning-workers`: label 必須（ユーザーが worker を見分けるため）と、
  spawn 返り値の model / effort 確認を追加。
- `quality-ops`: 旧 6 原則のうち task-intake / template / acceptance に移した
  ものを除き、横断規律 3 つ（同一ファイル直列化 / 統合レビュー / done = merge）に
  再構成。

### 互換性

既存ブロック名（role / responsibilities / no-investigate / running-workers /
spawning-workers / monitoring / lifecycle / worker-status / projects / tools /
model-policy / quality-ops / behavior）はすべて維持し、新設 3 ブロックは新名で
追加した。profiles の `prompt_blocks`（disable / override_blocks）による既存
カスタマイズはそのまま動く。カスタム `master-system.md` を使っている場合は
更新が届かないため、changes.yaml rev 5（guided）が prompt_blocks への移行を
対話で案内する。

## 二層配布の考え方

worker 側の品質ゲートは 2 経路で届ける（防御の多重化）。

1. **worker プロンプト経由（主経路）**: master が型の「検証手順 + 報告様式」を
   毎回埋めて送る。tako を入れた瞬間から、ユーザー設定ゼロで全 worker に効く。
2. **~/.claude/CLAUDE.md 経由（恒常層）**: setup のセクションテンプレ
   `06-completion-verification` を CLAUDE.md に取り込むと、worker を含む
   すべての claude セッションに常時効く。master がプロンプトを崩しても
   こちらが残る。既存ユーザーへは rev 5（guided）で同意の上追記。

## 運用知見 → ルールの対応（汎用化したもの）

| 観測された失敗（一般化） | 反映先 |
|---|---|
| 複数依頼の 1 worker まとめ投げで全部が粗くなる | task-intake Step 1–2 |
| 「ビルド成功」を「動く」と混同した完了報告 | template 検証手順 3 / 06 テンプレ |
| 完了報告を鵜呑みにして実は未実装だった | acceptance 2–3（証拠と diff） |
| ハード/UI 依存機能をビルド成功だけで「完了」と報告 | acceptance 4（未検証明示） |
| idle 通知の空振りで思考中の worker を潰しトークン二重消費 | monitoring（確認と待機、respawn 条件の閉じたリスト） |
| 長尺タスクが無コミットのまま中断して全損 | template Git 節 / lifecycle |
| 並行 worker が同一ファイルを触って先行修正を退行 | task-intake Step 3 / quality-ops 1 |
| 根因不明のまま「調べて直せ」で worker が迷走 | template 記入規則（根因先行 + 偵察） |
| 要件超過の凝った実装で課題・仕様ものが減点 | template 記入規則（要件密着） |
| 差し戻しの無限ループ（同じ指示で何度もやり直し） | acceptance 5（欠陥リスト + 2 回で方針再考） |
| worker 内のサブエージェント乱立で進捗が不可視化 | template Constraints 固定文 |

## 今後の候補（本 PR ではやらない）

- **worker への直接 system prompt 注入**: 現状 worker 規律はプロンプト本文で
  運んでいる。`build_worker_claude_cmd` に `--append-system-prompt-file` を足し、
  配布の worker-system テンプレを読ませれば、master のプロンプト品質に依存しない
  第 3 の層になる。spawn 経路（dispatch）に手が入るため、並行開発中の変更と
  衝突しないタイミングで別 Issue として起こす。
- master プロンプトの肥大監視: ブロックが増えたため、実測でトークン・遵守率に
  問題が出たら分割・圧縮を検討する。
