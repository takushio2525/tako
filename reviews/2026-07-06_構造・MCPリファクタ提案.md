# tako コードベースレビュー: 構造・MCP リファクタ提案

- 実施日: 2026-07-06
- 対象: 全 4 クレート（tako-core / tako-control / tako-app / tako-cli）約 44,000 行
- 方法: 主要ファイルの精読（dispatch.rs / mcp.rs / ipc.rs / remote.rs / orchestrator / protocol.rs /
  tako-cli main.rs / tako-app main.rs ほか）+ `cargo clippy --workspace --all-targets`（警告 0）+
  `cargo test --workspace`（342 passed / 5 ignored、全緑）+ GUI 操作と dispatch 対応の突き合わせ
- 本レビューはコード変更を伴わない。提案のみ

## サマリ

提案 17 件（高 5 / 中 7 / 低 5）。うち 1 件は**実バグ**（H1: `orchestrator_run` の
output が常に空）。高優先度 5 件は GitHub Issue に起票済み。

全体として層分離・テスト・コメントの品質は高く、大規模リファクタが必要な状態ではない。
問題が集中しているのは (1) **オーケストレーター完了待ちロジックの MCP / CLI 二重実装**
（既にバグと挙動乖離を生んでいる）、(2) **MCP HTTP サーバーの直列処理**、
(3) **UI 機能追加のたびに肥大する ControlHost trait** の 3 点。

## 良い点

- **3 層アーキテクチャの規律が実際に守られている**: 依存方向 tako-app → tako-control →
  tako-core が Cargo.toml レベルで強制され、GPUI 依存が tako-app に完全隔離されている。
  GPUI pre-1.0 の破壊的変更リスクへの防波堤として設計どおり機能している
- **dispatch 一元化**: UI / CLI / MCP が同一の `Request` → `dispatch()` を通る構造は
  設計原則 5「AI フルコントロール」の実装骨格として正しく、51 ツールという操作面の広さを
  1 箇所のセマンティクスで支えている
- **テスト文化**: 342 本の unit テスト + セルフテスト機構（`TAKO_SELF_TEST=1`）。
  日本語テスト名が仕様書として読める。IPC の fd リークテスト（`連続接続でfdが漏れない`）の
  ような非機能面のテストまである
- **clippy `-D warnings` 緑・`#[allow(dead_code)]` は 2 箇所のみ**。dead code がほぼない
- **コメントが「なぜ」を書いている**: Issue 番号・実測日・採用しなかった代替案への言及が
  多く、経緯の追跡可能性が高い（例: mcp.rs の Content-Length 固定の理由、CSI u の実測記録）
- **MCP instructions / ツール説明文への行動規範埋め込み**（FR-2.7.5）はプロダクトの
  差別化点として独自性がある

## 高優先度

### H1. [バグ] `orchestrator_run` の output が常に空文字列（MCP / CLI 両方）

- **対象**: `crates/tako-control/src/mcp.rs:1273`、`crates/tako-cli/src/main.rs:1443`
- **現状の問題**: worker 完了後の出力取得で `Request::Read` の応答から
  `result["content"]` を読んでいるが、dispatch の Read 応答は
  `{"pane": ..., "text": ...}`（dispatch.rs:484）。`content` キーは存在しないため、
  `tako_orchestrator_run` / `tako orchestrator run` の返す `output` は**常に空**。
  「spawn + 完了待ち + 出力取得 + close を 1 回で行うアトミック操作」という
  ツールの本旨（worker の成果を master に返す）が壊れている。auto_close 既定 true のため
  ペインは閉じられ、master は結果を読み直すこともできない
- **提案**: `result["text"]` への修正 + 「output が非空であること」の回帰テスト。
  同じ間違いが 2 箇所にあるのは H2 の重複が原因なので、H2 と併せて直すのが理想
- **期待効果**: orchestrator run が仕様どおり機能する
- **作業規模**: 小（2 行 + テスト 1 本）

### H2. [重複] オーケストレーター完了待ちループが MCP / CLI に二重実装され、挙動が既に乖離

- **対象**: `crates/tako-control/src/mcp.rs:1132-1301`（orchestrator_run 約 170 行）、
  `crates/tako-cli/src/main.rs:1138-1247`（orchestrator_watch）+ `1306-1467`
  （orchestrator_run 約 160 行）、判定ヘルパー `screen_looks_busy` / `screen_looks_idle` /
  `tail_lines` は mcp.rs と tako-cli の 2 箇所 + dispatch.rs の `is_worker_busy` にも同種実装
- **現状の問題**:
  - ポーリング状態機械（idle_streak / gone_streak / need_streak / 20 秒初期待機）が
    ほぼコピーで 2 系統ある
  - **既に挙動乖離**: CLI 版は `tmux_session_alive()` により tako 再起動中の gone を
    取り消すが、MCP 版にはこの防御がなく、tako 再起動時に `status=error` へ誤判定しうる
  - H1 のバグが 2 箇所で同時発生したのはこの重複の直接の帰結
- **提案**: tako-control に「完了待ちエンジン」を 1 本化する。
  `fn wait_for_worker(exec: &mut dyn FnMut(Request) -> Result<Value, String>, opts) -> RunOutcome`
  の形にすれば、MCP 側は `McpSession.exec`、CLI 側は `send_request` をそのまま注入できる。
  busy/idle 判定ヒューリスティックは既存の `claude_tui` モジュール（画面状態検出の正）へ集約
- **期待効果**: バグ修正が 1 箇所で済む・挙動統一・約 300 行削減
- **作業規模**: 中（ロジック移動 + 両呼び出し口の差し替え + テスト移設）

### H3. [性能] MCP HTTP サーバーが単一スレッド直列処理 — `orchestrator_run` が最大 30 分間、全 MCP 操作をブロック

- **対象**: `crates/tako-control/src/mcp.rs:1764-1780`（`McpServer::start` の
  `for request in server.incoming_requests()`）と `mcp.rs:1195`（20 秒 sleep）+
  ポーリングループ（既定タイムアウト 1800 秒）
- **現状の問題**: HTTP トランスポートは 1 スレッドでリクエストを直列処理する。
  通常ツールは dispatch 往復のみで高速だが、`tako_orchestrator_run` はハンドラスレッド上で
  完了までポーリングする設計（コメントにも「MCP ハンドラスレッドで合成する」と明記）。
  その間、**HTTP 経由（`TAKO_MCP_URL` 直結）の他エージェントの全ツール呼び出しが応答待ちで凍る**。
  stdio ブリッジ（`tako mcp serve`）は接続ごとに独立プロセス + IPC が接続毎スレッドなので
  影響しないが、HTTP 経路の並行性欠如はマルチエージェント運用（tako の核心ユースケース）と
  正面衝突する
- **提案**: 接続処理をスレッド化する（tiny_http の `Server` は `Arc` 共有 +
  複数スレッド `recv()` の並行 accept をサポートする設計。リクエスト毎
  `std::thread::spawn` でも、IPC 側と同じ接続毎スレッドパターンでよい）。
  併せて orchestrator_run の長時間ポーリングは専用スレッドで行う旨をコメントに残す
- **期待効果**: run 実行中も他エージェントの MCP 操作が通る。複数 master /
  複数 worker の並行運用が安全になる
- **作業規模**: 小〜中（スレッド化自体は数十行。token/tx の clone と送信順の確認）

### H4. [設計原則 5] タブ単位の退避（`shelve_tab`）が GUI からしか操作できない

- **対象**: `crates/tako-core/src/workspace.rs:392`（`shelve_tab`）、呼び出しは
  `crates/tako-app/src/main.rs:2643-`（タブ最小化ボタン）と TabDrag の D&D
  （drawer.rs:339 / status_bar.rs:100）のみ
- **現状の問題**: ペイン単位の退避は `Request::Background` + CLI `tako background` +
  MCP `tako_background_pane` で揃っているが、**タブ単位の一括退避は UI 専用**。
  AGENTS.md の開発不変条件「すべての機能は追加した時点で MCP / CLI から操作可能で
  なければならない。例外は理由を requirements.md に明記」に対し、例外記載も無い状態。
  なお今回の突き合わせで GUI 専用と確認できた操作は他に「バックグラウンドドロワーの
  表示トグル」「ペインのフォントズーム」「新規 OS ウィンドウ」（M2 参照）
- **提案**: `Request::Background` に `tab: Option<u64>` を追加（Split の pane/tab
  排他パターンを踏襲）。CLI は `tako background --tab N`、MCP は
  `tako_background_pane` に tab パラメータ追加
- **期待効果**: 開発不変条件の回復。AI が「このタブごと片付けて」に応えられる
- **作業規模**: 小（dispatch 分岐 + 引数 3 箇所 + テスト）

### H5. [保守] `ControlHost` trait の肥大（45 メソッド）— UI 機能追加のたびに際限なく伸びる構造

- **対象**: `crates/tako-control/src/dispatch.rs:29-204`
- **現状の問題**: dispatch が UI 状態へ触るための trait が、ワークスペース操作・
  セッション IO・パネル/ドロワー状態・プレビュー・ピン留め・動画再生・リモート・
  Chrome・アプリ更新まで 45 メソッドに達している。新機能 1 つごとに
  trait + TakoApp 実装 + テストホストの 3 箇所へメソッドが増える。既定実装で
  ボイラープレートは抑えているが、責務の境界がなく「dispatch が UI に求めるものの
  全集合」になっており、tako-app 側の実装（main.rs:5188-5598）も 400 行超
- **提案**: 責務別サブトレイトへ分割し、`ControlHost` はスーパートレイトとして合成する。
  例: `WorkspaceHost`（workspace/session/attach/detach）/ `UiStateHost`
  （panel/filetree/collapse/pin/drawer）/ `PreviewHost`（preview/video）/
  `SystemHost`（remote/update/chrome/cleanup）。dispatch の各アームが必要とする
  能力が型に現れ、テストホストは関係するトレイトだけ実装すればよくなる
- **期待効果**: 新機能追加時の接触面が局所化。テストホストの見通し改善。
  「どの操作がどの層の状態に触るか」が型で読める
- **作業規模**: 中〜大（機械的な移動だが接触面が広い。段階分割可）

## 中優先度

### M1. [重複] MCP ツール定義の三重管理（カタログ / build_request / CLI 出力整形）

- **対象**: `mcp.rs:138-1093`（`tools()` 手書き JSON 約 950 行）、
  `mcp.rs:1335-1646`（`build_request` の名前→Request 写像）、
  `tako-cli/src/main.rs:2186-`（`print_result` の Command 別整形）
- **現状の問題**: ツール 1 つの追加・変更で、スキーマ・写像・CLI・セルフテストの
  ツール数期待値（51 ハードコード）を人手で同期している。progress.md にも
  「ツール数セルフテスト期待値を修正」が繰り返し登場しており、同期コスト増が実測されている
- **提案**: ツール 1 件 = `{name, description, schema, build: fn(&Value, caller) -> Result<Request>}`
  の宣言テーブル（`Vec<ToolDef>` or 配列）へ統合し、`tools()` と `build_request` を
  テーブルから導出する。ツール数テストは `tools().len()` との自己整合でなく
  「名前一覧のスナップショット」にすると増減が diff で見える
- **期待効果**: 追加時の同期箇所が 1 箇所に。名前 typo による写像漏れがなくなる
- **作業規模**: 中（構造の組み替えのみ、挙動不変）

### M2. [設計原則 5] ドロワー表示トグル・ペインのフォントズームが dispatch 未対応

- **対象**: `status_bar.rs:94-97`（`drawer_visible` 直接トグル）、
  `main.rs:2547`（`zoom_focused_pane`、cmd+= / cmd+- / cmd+0）
- **現状の問題**: バックグラウンドドロワーの開閉と、ペイン単位フォントサイズは
  GUI からしか操作できない。`Request::Panel` は右パネルと filetree のみ対応。
  AI が「退避中の一覧をユーザーに見せる」「このペインの文字を大きくして見せる」を実行できない
- **提案**: `Request::Panel` に `drawer: Option<bool>` を追加。ズームは
  `Request::Zoom { pane, font_size: Option<f32> }`（None でリセット）を新設。
  OS ウィンドウ新規作成（cmd+shift+N）は AI 対象外の例外として requirements.md に明記
- **期待効果**: 開発不変条件の完全化。「見せる」導線の強化
- **作業規模**: 小

### M3. [性能] `sync_backend_windows` が UI スレッドで tmux サブプロセスを直列実行

- **対象**: `crates/tako-app/src/main.rs:2069-2110`、呼び出しは 2 秒ポーリングの
  適用ステップ（`this.update` 内、main.rs:1364 付近）と `refresh_tmux_data`
- **現状の問題**: 2+ window のバックエンドセッションがあるとき、非アクティブ window
  ごとに `capture_pane_text`（tmux 子プロセス起動）を **UI スレッド上で直列実行**する。
  オーケストレーター運用（子 worker が `tmux new-window` を作る形態）で window 数に
  比例して 2 秒ごとに UI がブロックする。2026-06-14 の「tmux ポーリング非同期化」で
  一覧取得は background 化済みだが、この window キャプチャが取り残されている
- **提案**: window キャプチャを background executor 側（`fetch_tmux_sessions` の
  コンテキストに backend_sessions を含めて一緒に取得）へ移し、UI スレッドは結果の
  適用のみにする
- **期待効果**: 描画のヒッチ解消（1 キャプチャ 5〜10ms × window 数）
- **作業規模**: 小〜中

### M4. [移植性] orchestrator / remote の保存先パスが macOS / /tmp ハードコード

- **対象**: `orchestrator/mod.rs:15-17`（`~/Library/Application Support/...` 固定）、
  `remote.rs:53-58`（state ファイルが `/tmp` 固定）
- **現状の問題**: tako-core には Linux（XDG）対応済みの `paths::data_dir()` が
  あるのに orchestrator は独自に macOS パスを組み立てており、Linux では
  `~/Library/...` を作ってしまう。remote の PID/token/port ファイルも `/tmp` 直で、
  Windows（Phase 6 必須）で全滅する。パス解決の正が 2 系統ある状態
- **提案**: `tako_core::paths::data_dir()` に一本化（`orchestrator/` サブディレクトリ、
  remote state は `<data_dir>/remote/`）。既存パスからの移行は初回起動時に
  ファイルがあれば移動（migrate_legacy と同パターン）
- **期待効果**: Phase 6 の移植作業からパス問題が消える。設定ファイルの置き場が一貫する
- **作業規模**: 小〜中（移行ロジック含む）

### M5. [構造] main.rs 9,801 行の分割続行 — セルフテスト約 3,100 行の分離が最も効く

- **対象**: `crates/tako-app/src/main.rs:6456-9611`（セルフテスト）、
  `main.rs:401-561`（TakoApp 約 70 フィールド）
- **現状の問題**: 2026-06-24 の 7 モジュール分割後も main.rs は最大ファイルのまま。
  内訳はセルフテストが約 1/3、TakoApp::new 約 650 行、render_pane 約 420 行、
  ControlHost 実装 400 行。TakoApp は 70 超フィールドの god struct で、
  roadmap.md にも「TakoApp 構造体の分割」「共通 UI ユーティリティの抽出」が未了で残っている
- **提案**: 第一歩として自己完結度が最も高いセルフテスト全体を `self_test.rs` へ移す
  （挙動不変・リスク最小・-32% 行数）。次に TakoApp のフィールドを
  ScrollState / DragState / VideoState / GitPanelState 等のサブ構造体へ段階的に畳む
- **期待効果**: main.rs が約 6,700 行へ。ビルド並列性・編集時の見通し改善
- **作業規模**: セルフテスト分離は小、サブ構造体化は中〜大（段階実施）

### M6. [型安全] `Request` の `action: String` フィールドを enum 化

- **対象**: `protocol.rs` の `Update` / `VideoPlayback` / `OrchestratorProjects` /
  `OrchestratorProfiles`（いずれも `action: String`）
- **現状の問題**: 不正な action が dispatch の実行時 match まで到達してからエラーになる。
  受理値の一覧が protocol（型）ではなく dispatch（実装）と MCP スキーマ（JSON）に散在
- **提案**: `#[serde(rename_all = "kebab-case")]` 等の enum へ置換（wire 表現は不変）。
  MCP スキーマの enum 列挙とデシリアライズ検証が一致する
- **期待効果**: 不正値はパース時点で弾かれ、網羅性チェックが働く
- **作業規模**: 小

### M7. [API 一貫性] `pane` / `pane_id` / tmux target の引数名不統一

- **対象**: `OrchestratorWorkerStatus { pane_id: u64 }`、
  `RemoteScrollback { pane_id: String }`（実体は tmux target "session:window.pane"）、
  他の全リクエストは `pane: Option<u64>`
- **現状の問題**: 同じ「pane_id」という名前で tako ペイン ID（u64）と tmux target
  文字列の 2 つの意味が混在する。特に `tako_remote_scrollback` は説明を読まないと
  tako ペイン ID を渡して失敗する（AI エージェントが最初に踏む典型的な罠）
- **提案**: RemoteScrollback の引数を `target`（または `tmux_pane`）へ改名し、
  `#[serde(alias = "pane_id")]` で後方互換を維持。OrchestratorWorkerStatus は
  `pane` への統一を検討（同じく alias で互換）
- **期待効果**: エージェントの誤用減。ツール群全体の引数語彙が 1 つに
- **作業規模**: 小

## 低優先度

### L1. [依存] webview PoC / 動画の依存を feature flag 化

- **対象**: tako-app の `tungstenite` / `data-encoding` / `ureq`（webview.rs = FR-3.8 PoC 用。
  ureq は update_checker と共用）、video_player.rs の AVFoundation FFI
- **現状の問題**: PoC 段階の機能の依存が常時ビルドに入る。Windows ビルド（CI）でも

- **期待効果**: PoC を捨てる/差し替える判断が容易になる。ビルド時間の微減

- **提案**: dispatch 結果が object のとき `structuredContent` を併記する（数行）
- **期待効果**: クライアントのパース堅牢性・将来の schema 検証対応
- **作業規模**: 小

### L3. [重複] 小物ユーティリティの散在

### L4. [MCP] ツールカタログの総量最適化

- **対象**: `tools()` 全体（51 ツール、シリアライズ済みで 32KB 超 — chunked 境界問題が
  顕在化した実績あり = セルフテストで確認済み）
- **現状の問題**: エージェント側のコンテキストをツール定義だけで大きく消費する。
  説明文は品質が高い一方、行動規範の重複記載（instructions と個別ツールの両方）など

- **提案**: 説明文の重複刈り込み + 使用頻度の低いツール（video / chrome / update 系）の
  説明短縮。ツール統合（例: video_playback + video_seek → tako_video）は

- **期待効果**: 全エージェントの毎セッションのトークン消費減
- **作業規模**: 小（文言）〜中（統合）

### L5. [テスト] UI 層モジュールの unit テスト不在

- **対象**: right_panel.rs（1,928 行）/ sidebar.rs（832 行）/ drawer.rs / status_bar.rs /

- **期待効果**: 退行検知の高速化（セルフテストは起動が重い）
- **作業規模**: 方針明文化は小、切り出しは継続的
