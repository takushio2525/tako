## 1. 結論

tako 本体は、Rust / GPUI を使ったネイティブ GUI ターミナルとして、全体的に設計品質が高い。
特に以下は強い。

- `tako-core` / `tako-control` / `tako-app` / `tako-cli` の依存分離
- UI / CLI / MCP が `Request` と dispatch を共有する制御プレーン
- 日本語テスト名を含む厚い回帰テスト
- ペイン内容・入力・`TAKO_TOKEN` をログへ出さない規約
- tmux 永続化、PTY、IME、OSC、MCP といった難所に対する実測ベースのコメント
- AI エージェントを「端末の利用者」ではなく「画面を操作する主体」として扱う独自性

一方、`tako remote` は現状のまま「安全なインターネット越しリモート端末」と断言できない。
短時間・本人利用・Cloudflare を信頼する前提のベータ機能としては成立するが、機密情報を扱う端末を
常時公開する用途には P0 対応が必要である。

### 総合評価

| 対象 | 評価 | コメント |
|---|---:|---|
| ローカルのターミナル本体 | A- | 難しい領域をよく制御している。巨大ファイルと UI 同期処理は要整理 |
| core / control の設計 | A | GPUI 依存隔離と dispatch 共有が有効 |
| テスト文化 | A- | unit / self-test は強い。remote の攻撃系・Worker 並行系が不足 |
| MCP / CLI 制御面 | B+ | 差別化点。ただし権限モデルと危険操作の承認境界が弱い |
| `tako remote` の認証トークン | B+ | 256-bit CSPRNG、fragment 化、定数時間比較は良い |
| `tako remote` の秘密管理 | C | `/tmp`、MCP への生トークン、PWA localStorage が問題 |
| `tako remote` のプロセス管理 | C | stop の終了確認、PID 同一性、tunnel 監視が不足 |
| relay / Pages / Tunnel の信頼設計 | C | Cloudflare/PWA が端末全権を持つ。KV の非原子性もある |
| Windows 対応 | C+ | ビルド意識はあるが named pipe、remote stop、実機確認が未完 |
| コンセプト | A- | agent cockpit として強い。IDE / remote / updater へのスコープ拡大に注意 |

## 2. `tako remote` のデータフローと信頼境界

```text
Mac / tako remote daemon
  ├─ 0.0.0.0:7749 の HTTP API
  ├─ bearer token
  ├─ tmux backend を直接操作
  └─ cloudflared Quick Tunnel
          │ HTTPS（Cloudflare edge で復号）
          ▼
    *.trycloudflare.com
          ▲
          │ machineId → tunnel URL
Cloudflare Workers KV relay
          ▲
          │ resolve
Cloudflare Pages PWA
          │ URL fragment から token 取得
          │ localStorage へ保存
          ├─ Authorization: Bearer token
          └─ WebSocket subprotocol: token.<TOKEN>
```

この経路では、次が trusted computing base に含まれる。

- tako daemon
- cloudflared 実行ファイル
- Cloudflare Tunnel / edge
- relay Worker と Workers KV
- Cloudflare Pages のデプロイ内容
- PWA の JavaScript と service worker
- スマホブラウザと localStorage
- bearer token が表示される QR / URL / CLI 画面

「HTTPS だから安全」だけではなく、Pages が配信する JavaScript 自体が bearer token を読み、
任意の terminal input を送れる点が重要である。

## 3. 既に正しく実装されている remote 対策

以下は問題ない、または良い防御である。

- remote は既定 OFF。明示的な start が必要
- token は 32-byte CSPRNG を hex 化した 256-bit 値
- daemon 起動ごとに token を再生成し、停止で無効化
- 接続 URL の token は URL fragment に置かれ、通常の HTTP request / Referer に送られない
- REST API は `/api/health` と静的ファイル以外 bearer token 必須
- WebSocket も `Sec-WebSocket-Protocol` の token で認証
- token 比較は内容について早期 return しない比較に変更済み
- `remote status` は既定で token と URL 内 token をマスク
- insecure モードは明示 opt-in で警告を表示
- relay Worker には端末 secret の SHA-256 による登録保護がある
- relay には terminal content や remote bearer token を送っていない
- PWA の terminal 描画は `textContent` / TextNode ベースで、確認範囲に `innerHTML` や
  `dangerouslySetInnerHTML` はない
- session ID は英数字とハイフンへ制限され、transcript の path traversal を防いでいる
- input text は shell 文字列連結ではなく tmux argv として渡している
- README には任意コマンド実行、token 共有禁止、Cloudflare での復号、insecure の平文性が記載されている

## 4. P0: remote 正式提供前に直すべき問題

### P0-1. secure モードでも LAN 全体へ平文 HTTP を bind している

- 場所: `crates/tako-control/src/remote.rs:126-130`
- 現状: secure / insecure に関係なく `0.0.0.0:<port>`
- cloudflared の接続先: `http://127.0.0.1:<port>`

既定 secure モードでも、同一 LAN から HTTP server 自体へ到達できる。
認証 token があるため即 takeover ではないが、README の「トンネル経由でのみホスト」と一致せず、
不要な攻撃面を残している。

#### 推奨

- secure: `127.0.0.1:<port>`
- insecure: `0.0.0.0:<port>`
- IPv6 を扱うなら secure は `::1`、insecure は明示的に `[::]`
- bind 先を status に表示
- secure daemon では LAN URL を生成しない

### P0-2. MCP / AI の tool result に生 token が入る

- 場所: `crates/tako-control/src/remote.rs:269-279`
- MCP 定義: `crates/tako-control/src/mcp.rs:970-1020`

`spawn_daemon()` の起動結果には以下が含まれる。

- `token`
- token 入り `connect_url`
- token 入り `fallback_url`

MCP には `tako_remote_start` と `tako_remote_status(show_token=true)` があり、AI は remote secret を
直接受け取れる。tool result はモデルコンテキストや Claude transcript に残りうる。

また AI は `insecure=true` を指定できるため、prompt injection や誤判断で平文 remote を開く可能性がある。

#### 推奨

- MCP から `show_token` を削除
- MCP の `insecure=true` を禁止
- remote start はローカル UI の確認ダイアログを必須化
- MCP 応答は `{running, port, pairing_displayed}` 等の非秘密情報だけにする
- QR / 接続 URL は tako のローカル UI にだけ表示
- AI フルコントロールの例外として `.agent/requirements.md` に理由を明記

### P0-3. 共有 `/tmp` の予測可能な state ファイル

- 場所: `crates/tako-control/src/remote.rs:69-112`
- 対象: PID / token / port / tunnel state
- QR: `<temp_dir>/tako-remote-qr.png`

固定名のファイルを共有 `/tmp` に書いている。token と QR は書き込み後に chmod 0600 されるが、
作成時点では umask 依存であり、既存 symlink も辿る。

共有 Mac では別ユーザーが先に symlink を置き、攻撃者所有の書き込み可能ファイルへ token を書かせる
余地がある。複数 OS ユーザーや複数 remote instance の衝突も起きる。

#### 推奨

- `<data_dir>/remote/` を 0700 で作成
- token / state / QR をその配下へ移動
- 作成時から 0600（OpenOptionsExt::mode）
- `O_NOFOLLOW`、`create_new`、temp + atomic rename
- instance nonce を持つ state directory
- QR はランダム名にして停止時に削除

### P0-4. remote stop が停止成功を検証しない

- 場所: `crates/tako-control/src/remote.rs:452-477`

現在は PID へ SIGTERM を送り、kill の戻り値を無視し、500ms 後に state files を消して成功を返す。

想定される問題:

- SIGTERM 失敗でも `stopped: true`
- daemon / cloudflared が残ったまま管理情報だけ消える
- PID 再利用時に無関係な同一ユーザープロセスを対象にする
- daemon が残ると既存 token で remote access が継続する

#### 推奨

- PID、実行ファイル、起動時刻、instance nonce を照合
- SIGTERM の戻り値を確認
- 一定時間 `waitpid` / process probe
- 終了しなければエラーを返し state を残す
- 明示的な `--force` のみ SIGKILL
- daemon 自身の認証付き control socket から shutdown する方式を優先

### P0-5. 公開 Pages PWA が端末への全権を持つ

- token 読み込み: `web/tako-remote/src/pages/connect.jsx:26-45`
- token 永続化: `web/tako-remote/src/store.js:1-31`
- API 使用: `web/tako-remote/src/api.js:4-75`

Pages origin の JavaScript は token を読み、terminal screen、transcript、任意 input を扱う。
次の侵害が remote shell compromise に直結する。

- Cloudflare account / Pages deployment
- release script / CI credential
- PWA supply chain
- PWA の XSS
- Cloudflare edge または配信内容の改変
- service worker の悪意ある更新

README の「E2E ではない」は正しいが、「Cloudflare が通信を復号する」に加えて、
「Pages から配信される JavaScript 自体が接続鍵を扱う」ことも説明すべきである。

#### 推奨

- CSP、`frame-ancestors 'none'`、`object-src 'none'`、`base-uri 'none'`
- `Cache-Control: no-store` を screen / transcript / agents API へ付与
- `Referrer-Policy: no-referrer`
- Google Fonts 等の外部資産を自己ホスト
- localStorage ではなく短期 session、または端末鍵による保護
- remote 稼働中は Mac 側 UI に常時表示
- 高セキュリティ用途は native companion client または署名済みクライアントを検討

## 5. High / Medium の remote 問題

### H-1. relay の first-write-wins は原子的でない

- 場所: `web/tako-remote-worker/src/index.js:108-133`

Workers KV の `get(secretKey)` → `put(secretKey)` → `put(machine)` は transaction ではない。
初回の同時登録では複数 request が「secret 未登録」と判断できる。

レート制限も KV の read-modify-write なので、並列 request が同じ current value を読み、上限を回避できる。
同一キーの高頻度 write 自体が Workers KV の制限に触れる可能性もある。

Cloudflare 公式資料:

- https://developers.cloudflare.com/kv/api/write-key-value-pairs/

#### 推奨

- Durable Objects で machineId 単位に直列化
- Cloudflare Rate Limiting / WAF を利用
- registration payload を端末鍵で署名
- secret hash 比較と mapping update を同一の直列化された主体で処理

### H-2. tunnel URL の検証が substring 判定

- 場所: `web/tako-remote-worker/src/index.js:98-101`

現在は `startsWith('https://') && includes('.trycloudflare.com')` なので、例えば以下も通る。

```text
https://foo.trycloudflare.com.attacker.example
https://attacker.example/?next=.trycloudflare.com
```

#### 推奨

```js
const u = new URL(tunnelUrl);
const ok = u.protocol === 'https:' &&
  u.username === '' &&
  u.password === '' &&
  /^[a-z0-9-]+\.trycloudflare\.com$/i.test(u.hostname) &&
  (u.port === '' || u.port === '443') &&
  u.pathname === '/' &&
  u.search === '' &&
  u.hash === '';
```

### H-3. 接続確認が未認証 health のみ

- 場所: `web/tako-remote/src/pages/connect.jsx:42-53`

PWA は `/api/health` が成功すると、token が正しいかを確認せず host / token を保存して接続完了にする。
relay が偽 host を返した場合、その host は health だけ実装すればよく、その後の `/api/panes` で
Authorization token を受け取れる。

#### 推奨

- bearer 必須の `/api/whoami` を追加
- daemon instance ID と machine ID を返す
- challenge nonce へ端末鍵で署名する
- health と認証成立を分離

### H-4. Quick Tunnel は production 用ではない

tako は TryCloudflare Quick Tunnel を既定経路にしているが、Cloudflare は testing / development only、
SLA なし、production では named tunnel を使うよう明記している。

- https://developers.cloudflare.com/cloudflare-one/networks/connectors/cloudflare-tunnel/do-more-with-tunnels/trycloudflare/

#### 推奨

- Quick Tunnel を「簡易 / beta」経路として明示
- named Cloudflare Tunnel + Access
- Tailscale / Headscale
- SSH forwarding
- ユーザー管理 reverse proxy
- LAN-only

を transport provider として選択可能にする。

### H-5. tmux output の pipe deadlock

- 場所: `crates/tako-control/src/remote.rs:828-882`

stdout / stderr を pipe にしたまま child exit を `try_wait()` で待ち、終了後に read している。
`capture-pane` が pipe buffer を超えると、child は書き込み待ちで exit できず、親は child の exit を待つ。

WS init は最大 2000 行の ANSI content を取得するため、通常利用でも pipe buffer を超えうる。

#### 推奨

- stdout / stderr を別 thread で同時 drain
- process timeout crate または wait thread + channel
- response size 上限
- 可能なら tmux control mode で subprocess 生成回数も削減

### H-6. timeout 実装が blocking read を止められない

- cloudflared stderr: `parse_tunnel_url()`
- daemon stdout: `spawn_daemon()`
- relay curl: `register_relay()`

`BufRead::lines()` は次の改行まで block するため、deadline の比較まで戻らない。
curl にも connect / total timeout がない。

#### 推奨

- reader thread + `recv_timeout`
- child kill + wait による確実な cleanup
- curl をやめ Rust HTTP client を共有、または `--connect-timeout` / `--max-time`

### H-7. remote だけ core / dispatch を迂回する

- 場所: `crates/tako-control/src/remote.rs:1-5`, `1607-1849`

remote は tmux を直接操作するため、次を通らない。

- 最後の pane を閉じない保護
- workspace / layout 更新
- `CloseReason` の整合
- Claude TUI の bracketed paste と Enter 送達確認
- pane / tab policy
- 操作 origin と監査

これは AGENTS.md の「操作ロジックは core API → dispatch へ 1:1」の原則とも不一致。

#### 推奨

- daemon から認証済み IPC で app dispatch を呼ぶ
- app 不在時の tmux read-only fallback だけを限定的に残す
- close / input / resize は core + dispatch の正規経路に統合

### H-8. Enter の失敗 status を確認していない

- 場所: `crates/tako-control/src/remote.rs:1072-1088`

本文送信の tmux status は確認するが、Enter 用の2回目は command 起動成功しか確認しない。
tmux が non-zero exit でも remote API は成功を返しうる。

PWA 側は input error を空 catch し、送信前に input field を消すため、ユーザーの command が失われる。

### M-1. remote token に TTL / idle timeout がない

daemon を止めるまで bearer token が有効で、公開 tunnel も残る。

#### 推奨

- 既定 1〜4 時間の有効期限
- idle timeout
- 起動時に期間選択
- status bar の残り時間
- 延長には Mac 側承認

### M-2. 接続端末・権限・revocation がない

単一 bearer token が全操作を許可する。

推奨 role:

- `observe`: screen / state のみ
- `interact`: input 可
- `manage`: close / resize 可
- `admin`: pairing / token revoke

### M-3. remote action の監査がない

terminal text や pane content はログ禁止のままでよいが、以下の metadata は記録可能。

- connection start / end
- device ID
- source IP の短期ハッシュ
- route と result status
- pane ID
- input byte count（内容なし）
- close / destructive action

### M-4. API response に no-store がない

screen、scrollback、agents、transcript は機密性が高い。ブラウザや proxy の cache に依存せず、
明示的に `Cache-Control: no-store, private` を付けるべきである。

### M-5. WebSocket connection ごとに 250ms 間隔で tmux subprocess

1 connection あたり毎秒約4回、変化時は追加 capture が発生する。複数 client / reconnect storm では
subprocess 数が増える。

#### 推奨

- pane ごとの共有 broadcaster
- 一つの capture 結果を複数 client へ配信
- 無変化時は polling interval を段階的に延長
- connection 数・pane ごとの subscription 上限

### M-6. relay secret を curl argv に載せる

`curl -d <JSON>` の argv に relay secret が含まれる。OS や監視ツールによっては process list から見える。
stdin で body を渡すか Rust HTTP client を使うべきである。

## 6. remote 以外のコードベースレビュー

### 良い点

1. **GPUI 隔離**
   - `tako-core` と `tako-control` が GPUI 非依存
   - pre-1.0 GUI framework の変更を app 層で吸収できる

2. **dispatch 一元化**
   - CLI と MCP の操作意味論を共有
   - protocol が操作カタログの中心として機能

3. **テスト文化**
   - pane tree、workspace、layout、IPC、MCP、PTY、OSC、tmux、orchestrator を広くテスト
   - fd leak、性能計測、実 CLI / Claude の e2e を持つ

4. **エラー処理の改善履歴**
   - PTY spawn failure で panic しない
   - close reason の区別
   - tmux / process timeout の意識

5. **コメント品質**
   - Issue、採用理由、実測結果、過去事故が残されている

### 問題・最適化候補

#### G-1. 大型ファイルへの責務集中

調査時点のおおよその規模:

- `tako-app/src/main.rs`: 約10,000行
- `tako-control/src/dispatch.rs`: 約4,000行
- `tako-control/src/mcp.rs`: 約2,500行
- `tako-cli/src/main.rs`: 約2,500行
- `tako-control/src/remote.rs`: 約2,400行

feature 追加時の変更衝突、レビュー負荷、意図しない coupling が増える。

推奨分割:

- dispatch: workspace / terminal / preview / orchestrator / system
- MCP: catalog / argument parsing / engine / HTTP transport
- remote: daemon state / tunnel / relay / auth / REST / WS / tmux adapter
- app main: lifecycle / workspace controller / terminal controller / diagnostics

#### G-2. `ControlHost` の肥大

UI state、workspace、preview、remote、update、system operation が一つの trait に集まっている。
責務別 subtrait または service object へ分けるとテスト host も軽くなる。

#### G-3. UI thread を塞ぐ同期 dispatch

既に GitLog / GitDiff で秒単位の UI 専有が観測されている。外部 process、filesystem scan、
large serialization は原則 background executor へ移し、UI には結果だけ返すべきである。

#### G-4. MCP tool 定義の重複管理

tool schema、tool name → Request、CLI argument、result formatting が別々に手書きされている。
tool 数が50を超えると追加漏れ・schema 乖離・期待件数の stale が起きやすい。

宣言的な operation registry または macro / typed schema 生成を検討する価値がある。

#### G-5. 危険操作の permission model がない

AI フルコントロールにより、MCP から update、file operation、remote start、close、send が可能。
機能の有無とは別に、次の risk class が必要である。

- read-only
- local reversible
- local destructive
- network exposure
- secret disclosure
- application update / external mutation

network exposure と secret disclosure は UI confirmation 必須にするべきである。

#### G-6. Windows の必須要件とのギャップ

- Unix domain socket の named pipe 版が未実装
- remote stop が Windows 未実装
- data_dir が Windows で未実装部分あり
- ConPTY / IME / font / process lifecycle の実機確認が未完
- CI 停止中のため現在の Windows 回帰を検出できない

Windows 対応を差別化軸として維持するなら、機能追加速度より先に CI と最小実機 smoke を復旧すべきである。

#### G-7. CI の security / web coverage が不足

`.github/workflows/ci.yml` は Rust build / fmt / clippy / test が中心で、以下が含まれていない。

- `cargo audit`
- remote PWA `npm ci && npm run build`
- relay Worker `npm test`
- `npm audit`
- release asset の secret scan / SBOM

#### G-8. parser / protocol fuzzing を追加する価値が高い

対象候補:

- OSC tap scanner
- ANSI parser
- layout.json restore
- URL / query parser
- tmux snapshot parser
- MCP / IPC JSON envelopes
- remote REST routing
- transcript JSONL normalizer

## 7. 依存関係監査

## 8. コンセプト評価

### 強い点

tako の本質は「AI エージェント向け terminal workspace / control plane」である。

- agent / worker / dev server / logs を1グループに集約
- AI が pane と layout を操作
- AI が成果物を開いて見せる
- MCP / CLI / passive detection を同居
- local-first OSS
- Electron ではなく native Rust UI

これは単なる terminal emulator との差別化として明確である。

特に「AI が作業するだけでなく、人間に見せるところまで行う」は良いプロダクト原則である。

### リスク

現在は以下を同時に内包している。

- terminal emulator
- lightweight IDE
- tmux persistence manager
- MCP control plane
- orchestrator
- setup / updater
- Chrome web view
- remote terminal
- public relay / public PWA

すべてを深く作ると「高速で軽い agent cockpit」より「独自 IDE / remote platform」に近づき、
保守コストと攻撃面が急増する。

優先順位は次がよい。

1. agent monitoring / attention management
2. control plane reliability
3. terminal quality / persistence
4.成果物 presentation
5. remote は安全な別境界として限定提供
6. editor / browser は軽い確認用途に留める

### ドキュメント上の矛盾

- `.agent/concept.md`: クラウド同期・アカウント機能は Non-goal
- `.agent/architecture.md`: リモート接続機能は持たない
- 現実: public Pages、public relay、Cloudflare Tunnel を含む remote feature が存在

remote を正式な柱にするなら、次を仕様化すべきである。

- threat model
- trusted parties
- Cloudflare へ送る metadata
- E2E ではないこと
- token / device / expiration
- abuse / incident response
- public relay の SLA と終了方針
- self-hosting path

## 9. 追加機能の提案

### 最優先: remote security / operation

1. **端末別ペアリング**
   - one-time QR
   - device public key
   - Mac 側で端末名と権限を承認

2. **read-only remote**
   - screen と状態だけ
   - input は追加承認

3. **端末別 revoke**
   - token 全交換ではなく1台だけ解除

4. **自動停止 / idle timeout**
   - 1h / 4h / until manually stopped

5. **remote kill switch**
   - status bar / menu bar から全接続即切断

6. **接続通知**
   - macOS notification
   - 接続端末、権限、最終操作

7. **transport provider**
   - Quick Tunnel beta
   - named Cloudflare Tunnel + Access
   - Tailscale
   - SSH
   - custom HTTPS URL

### agent cockpit として価値が高い機能

8. **Attention Inbox**
   - needs input / failed / review requested / completed を全 tab 横断で集約

9. **Agent Timeline**
   - spawn、tool use、質問、完了、失敗、close を時系列表示

10. **Resource Budget**
    - worker 数、CPU、memory、同時 process、token budget の上限

11. **Artifact Inbox**
    - worker が生成した diff / image / preview / URL を「成果物」として一覧化

12. **Session Snapshot**
    - layout、cwd、agent session、tmux session、uncommitted diff の復元ポイント

13. **Policy Profiles**
    - solo / trusted repo / untrusted repo / remote read-only など権限 preset

14. **操作履歴と Undo**
    - pane move / background / close 前 snapshot
    - destructive file action の undo

15. **Performance Dashboard**
    - UI stall、dispatch latency、tmux subprocess 数、render element 数

## 10. 推奨対応ロードマップ

### P0-A: remote の直近封じ込め

- secure は localhost bind
- state を user-private directory へ移動
- MCP から token / insecure を除去
- stop を終了確認方式へ
- API に no-store / security headers
- PWA の authenticated health

### P0-B: relay / Pages

- URL hostname 検証修正
- KV first-write / rate limit を Durable Objects または edge rule へ
- Pages CSP
- external fonts 自己ホスト
- Quick Tunnel の beta 表示

### P1: 正規アーキテクチャへの統合

- remote input / close / resize を dispatch 経路へ
- device pairing / permission role
- TTL / idle timeout / audit metadata
- shared WS broadcaster

### P2: プロジェクト全体

- main / dispatch / MCP / remote 分割
- CI security + web jobs
- RustSec 更新
- Windows CI / named pipe / actual-device smoke
- parser fuzzing

## 13. 最終判断

### tako 本体

継続開発に値する強いコンセプトと良い技術基盤がある。全面的な作り直しは不要。
必要なのは feature velocity を落とすことではなく、巨大モジュールの分割、危険操作の権限境界、
Windows / CI の継続保証を並行して進めることである。

### tako remote

現状は「本人が短時間使う beta remote」として明確に位置づけるべきである。
P0 対応前に、企業・機密リポジトリ・常時接続・第三者提供へ安全性を訴求するのは推奨しない。

最低限、次の4点が入れば大きく改善する。

1. secure localhost bind
2. MCP / model から remote secret を隔離
3. `/tmp` state 廃止 + 確実な stop
4. authenticated pairing + device / TTL / revoke

これらを満たしたうえで relay / Pages の信頼モデルを仕様として明文化すれば、tako remote は
「多数の AI agent を外出先から監視し、必要なときだけ介入する」という tako のコンセプトと強く噛み合う。
