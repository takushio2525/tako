# 開発ルール

## 言語

- 回答・コミットメッセージ・ドキュメント・コードコメントは日本語で記述する
- 技術用語とコード識別子は原語のまま維持する

## 対話スタイル

- 要点から先に伝え、短い指示は文脈から妥当な解釈で進める
- 大きな変更や意図が分岐する変更は、着手前に方針を提示する
- 前提のずれ、波及、落とし穴に気づいたら先に伝える

## Git 運用

- 機能単位でコミットし、無関係な変更を同じコミットへ混ぜない
- 影響が大きい変更はブランチと PR を使う
- コミット前に差分を読み、秘匿情報や個人情報がないことを確認する

## コード品質

- 依頼された範囲だけを変更し、無関係なリファクタや機能追加を行わない
- 新しい依存を追加する前に、既存の依存で実現できないか確認する
- コメントは理由が非自明な箇所を中心に書く

## 安全ルール

- 本番環境のデータを直接操作しない
- 破壊的操作の前に影響を確認する
- 認証情報・API キー・個人情報をコード、ログ、コミットへ含めない

## 提案の品質

- 技術選定や設計方針が分岐するときは、候補と推奨理由、トレードオフを示す

## 完了検証

- 完了報告前に build・lint・test と変更機能の実動作を確認する
- エッジケースを確認し、自分の diff を最初から最後まで読み直す
- 完了報告には実施した検証と未検証項目を明記する
