# 完了検証（品質ゲート）

## 必須概念

- 「完了」と報告する前に必ず通す自己検証手順を定義する
- 検証の証拠（実行したコマンドと実際の結果）を添えて報告するルールを定義する
- 検証できなかった項目を「未検証」として明示するルールを定義する

## 参考テンプレート

### 標準（全レベルで含めるべき）

```markdown
## 完了検証（品質ゲート）

「完了しました」と報告する前に、必ず以下をすべて実施する:

1. ビルド・リント・テストをすべて実行し、緑であることを確認する
2. 変更した機能を実際に動かし、期待どおり動くことを自分で観察する
   （**ビルドが通った ≠ 動く**。動作確認までがタスク）
3. エッジケース（空入力・エラー経路・境界値）を最低 2 つ試す
4. 自分の diff を最初から最後まで読み直し、デバッグ残骸・無関係な変更・
   リネーム漏れ・参照切れがないか確認する

完了報告には以下を含める:
- 何をどう変えたか（ファイルと要約）
- 検証の証拠（実行したコマンドと実際の出力の要点）
- 検証できなかったこと（実機が必要等）とその理由
```

### 厳格（チーム・OSS 向けの追加）

```markdown
- テストで担保できない変更（UI の見た目・実機依存の挙動）は、操作ログか
  スクリーンショットを残す。無い場合は「未検証」と明記して報告する
- 「たぶん動く」「〜のはず」で完了報告しない。証拠を示せない項目は
  完了に含めない
- 途中で詰まったら、黙ってスコープを変えずに、何を試してなぜ詰まったかを
  報告して指示を仰ぐ
```
